Composer

Zakaj composer?

  • Standardni problemi glede deljene uporabe kode
  • Update mehanizem - vedno najnovejša koda
  • Odprtokodni projekt z dobro dokumentacijo

* tako npm kot rubygems sta daleč naprej po nastanku in uporabi

  • Roll your own packages (composer-sentinel, packagist, ...)

Composer paket kot API: v praksi

  • JSON SHEMA je edini pravi vir podatkov
  • Generiramo:
    1. Testno konzolo (api.rtvslo.si/console)
    2. Dokumentacijo (api.rtvslo.si/console/docs)
    3. Dejanski primeri v dokumentaciji so generirani in ne fikcija
    4. PHP "routing" in interface za implementacijo, docbook komentarje
    5. Javascript odjemalce za razne projekte
  • Interna uporaba APIjev je identična ostalim composer paketom

Composer paket kot API

  • Pomankljivosti zaradi PHPja
    • Ne podpiramo korektnih HTTP verb apijev (GET/POST)
    • Ne delamo globalnega verzioniranja (v1, v2,...)
  • Prednosti HTTP glede na PHP uporabo
    • Avtomatski "tracking" API klicev (Client ID)
    • Statisticna obdelava klicev (stevilo, hitrost, vir)
      • api.rtvslo.si/status

Composer paket kot API

  • Napredno stanje
    • Nginx + LUA + Redis (klici <1ms)
    • Nekateri klici izključno PHP server-side (mail/create)
    • Nekateri klici imajo ločeno 100% LUA implementacijo
    • Scaling je skoraj 0 effort (vse je v composer paketu!)
